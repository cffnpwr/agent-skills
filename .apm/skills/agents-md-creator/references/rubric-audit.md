# ルーブリック監査チェックリスト

既存のAGENTS.md（またはCLAUDE.md）を改善する際の監査観点。各観点を評価し、問題があれば**差分提案**としてユーザーに提示する。修正のうちコードから読めない意図に関わる部分はユーザーに確認する。

## 観点1: 推奨骨格の網羅

- [recommended-skeleton.md](recommended-skeleton.md) の推奨セクションのうち、**該当するのに欠けているもの**がないか。
- 逆に、**該当しないのに残っているセクション**（空セクション、「なし」だけのセクション）がないか → 削除を提案。

## 観点2: 段階的開示の遵守

- 本体に**特定タスク時のみ必要な詳細**（込み入った手順、具体例、状況依存ガイド）が混入していないか → `.agents/docs/` への切り出しを提案。
- `.agents/docs/` に**常時適用すべき原則・規約**が埋もれていないか → 本体への昇格を提案。
- 判定は [progressive-disclosure.md](progressive-disclosure.md) の「常に読むべきか」軸で行う。

## 観点3: リポジトリ実態とのドリフト

- 記述が**古い・不正確**になっていないか。[content-gathering.md](content-gathering.md) の解析でリポジトリ実態を読み取り、既存記述と突き合わせる。
- 主なドリフト例:
  - 記載されたビルド・テスト・lintコマンドが現在のCI設定や `package.json` scripts と一致しない。
  - ディレクトリ構成の記述が実際のツリーと食い違う。
  - 言語・フレームワーク・依存のバージョンが古い。
  - もう存在しない規約やツールへの言及。

## 観点4: プレーン参照の参照切れ

- 本体から `.agents/docs/*.md` への参照先ファイルが**実在するか**。
- 参照切れがあれば、参照の削除か対象ファイルの作成を提案。
- 機械的にチェックしたい場合は [check_references.sh](../scripts/check_references.sh) を使ってよい。

## 観点5: CLAUDE.md symlinkの正否

- `CLAUDE.md` が存在する場合、それが `AGENTS.md` への正しいsymlinkになっているか、または実体ファイルとして内容が二重管理されていないか。
- 判定基準と提案手順は [claude-md-entrypoint.md](claude-md-entrypoint.md) を参照。
- `AGENTS.md` 本体に `@`参照がある場合、それが「常時コンテキストに読み込ませたい」という設計意図に沿った意図的なものか確認する。
  意図的なら許容する。必要時のみ読めばよい情報を `@`参照しているなら、プレーン参照（段階的開示）への変更を提案する。意図しない混入であれば除去を提案する。

## 観点6: 本体の肥大化

- 本体が常時適用の情報だけで構成されているか。肥大化していれば、観点2に基づき `.agents/docs/` への切り出しを提案。

## 提案のまとめ方

- 各観点の問題点と修正方針を整理してユーザーに提示する。
- 意図に関わる修正（禁止事項・設計方針の変更等）は勝手に決めず確認する。
- 事実のドリフト修正（コマンド・構成の更新等）はコード解析で裏が取れているものを提案する。
