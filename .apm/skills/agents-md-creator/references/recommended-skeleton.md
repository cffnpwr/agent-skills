# 推奨骨格の定義

AGENTS.md本体に置く推奨セクション群と、各セクションの採否判定基準。**該当しないセクションは出さない**（空セクションや「なし」を残さない）。順序は推奨であり、リポジトリ実態に応じて調整してよい。

本体に置くのは「どのタスクでも常時適用される情報」だけ。詳細手順・具体例・状況依存ガイドは `.agents/docs/` に切り出す（[progressive-disclosure.md](progressive-disclosure.md) 参照）。

## セクション一覧と採否基準

### 1. プロジェクト概要

- **内容**: 何のプロジェクトか、主要技術スタック、エージェントが最初に把握すべき全体像を数行で。
- **採否**: 原則として常に入れる。
- **収集元**: README、リポジトリ構成、依存定義（package.json・Cargo.toml・pyproject.toml 等）。

### 2. ビルド・テスト・lint・フォーマットコマンド

- **内容**: エージェントが実行すべき具体的コマンド（ビルド、テスト実行、lint、フォーマット、型チェック等）。
- **採否**: 該当するコマンドが存在する場合に入れる。存在しない種類は項目ごと省く。
- **収集元**: package.json scripts、Makefile、justfile、Taskfile、CI設定（.github/workflows 等）、各言語のビルドツール設定。
- **注意**: 常時適用の代表コマンドのみ本体に。込み入った手順や環境別の差異は `.agents/docs/build.md` 等へ。

### 3. ディレクトリ構成

- **内容**: 主要ディレクトリの役割。エージェントがどこに何があるかを把握するための地図。
- **採否**: 構成が自明でない、または把握に価値がある場合に入れる。
- **収集元**: 実際のディレクトリツリー。

### 4. コーディング規約

- **内容**: 命名規則、フォーマット方針、言語・フレームワーク固有の約束事、レビューで守るべきルール。
- **採否**: リポジトリに既存の規約（lint/formatter設定、既存コードの傾向、ドキュメント化された規約）がある場合に入れる。
- **収集元**: linter/formatter設定（eslint・prettier・ruff・rustfmt 等）、EditorConfig、既存コードの傾向、CONTRIBUTING。
- **注意**: 設定ファイルで機械的に強制される規約は「設定に従う」と一言で済ませ、設定で表現できない人的な約束事を中心に書く。
- **記述上の注意**: スコープ付きパッケージ名（`@types/node`、`@eslint/js` 等）に言及する際はバッククォートで囲う。誤読防止と検証ノイズ（`@`参照との混同）防止のため。

### 5. 禁止事項・制約

- **内容**: やってはいけないこと、触ってはいけない領域、必ず守る制約（例: 特定ディレクトリを編集しない、特定コマンドを使わない、破壊的操作の禁止）。
- **採否**: 制約が存在する場合に入れる。これはコードから読めない**意図**が中心なのでユーザー確認が必要。
- **収集元**: ユーザーへの確認が主。CI/設定から推測できるものは下書きに含めて確認する。

### 6. その他の状況依存セクション（任意）

リポジトリ実態に応じて以下を検討する。該当しなければ出さない。

- 開発環境セットアップのヒント（常時適用の最小限のみ。詳細は `.agents/docs` へ）。
- コミット・PR・ブランチ運用規約。
- セキュリティ上の考慮事項。
- VCS運用方針（git/jj、ブランチ戦略等）。

## 適応の原則

- 推奨骨格は出発点。リポジトリに該当しないセクションは**作らない**。
- 1つのセクションに常時適用情報と状況依存情報が混在する場合、常時適用分だけを本体に残し、残りを `.agents/docs` へプレーン参照で誘導する。
- セクション名・粒度はユーザーの使用言語とプロジェクトの慣習に合わせてよい。
