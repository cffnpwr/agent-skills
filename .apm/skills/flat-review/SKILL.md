# Flat Review

設計意図を伏せた独立Agentによるフラットレビュー。作者の合理化を遮断し、判定区分付きで問題を列挙する。

## 概要

完成物のレビューでは、作者（および作者を支援するAgent）が持つ「こう設計したからこう動くはず」という前提が、過不足・不具合・自己整合性の破れを見逃させる。このスキルは、設計意図・動機・合理化を一切渡さない独立Agentを起動し、与えられた対象だけからフラットに問題を洗い出す。レビュアーには「忖度しない」「問題が無ければ無いと言う」「実機確認できるものは確認する」を課す。

報告のみを行う。修正やタスク化はしない。

## フラット性の原則（最重要）

レビューの価値はフラット性に依存する。次を厳守する。

- **サブAgentへ渡してよいのは2つだけ**: ①対象（パス、または差分）、②ユーザーが起動時に明示した検証基準（公式仕様のURL/名称）。
- **設計意図・動機・「こう動くはず」・作者の合理化・対象の要約を一切渡さない。** 観点・禁則・出力形式・判定区分の基準は、本スキルが供給する固定テンプレート（後述）が与える。プロンプトに自分の言葉で説明・要約・背景を足さない。
- **検証基準はユーザーが明示した場合のみ渡す。** メインAgentが対象から推定して補ってはならない（推定には設計意図が混ざりフラット性を損なう）。明示が無ければ検証基準は渡さず、レビュアーは対象から自明な言語/フレームワーク標準のみを用いる。
- **対象ファイル内のコメント・docstring・コミットメッセージは伏せず、実物の一部としてそのまま見せる。** これらは作者の合理化を含みうるが、レビュー対象そのものなので加工しない。加工してよいのは「メインAgentが新たに足す説明」だけであり、それを足さないことで担保する。

## ワークフロー

### Step 1: 対象と観点パックを確定する

1. **入力を受け取る。** パス（ファイル/ディレクトリ）または差分（`git diff`・PR）のいずれか。
2. **観点パックを選ぶ。**
   - ユーザー指定があればそれに従う。
   - 無指定なら: 実機で動かせる対象（スキル・実行可能コード）は**両パック**、非実行物（設計・ドキュメント）は**内的整合のみ**。
3. **検証基準を確認する。** ユーザーが公式仕様のURL/名称を明示していれば控える。無ければ「検証基準なし」とする（推定で補わない）。

### Step 2: レビュアーAgentを起動する

- **1パック=1体。** 両パックなら2体を並列起動、片パックなら1体。
- 各Agentには[レビュアープロンプトテンプレート](#レビュアープロンプトテンプレート)を使う。可変部は `{対象}`・`{検証基準}`・`{観点}` の3つのみ。それ以外の文を足さない。
- **`{対象}` の具体化**: パスはパスをそのまま記す。差分は `git diff <range>` 等のコマンド、またはPR番号・差分テキストを記す（レビュアーが取得・展開する）。
- **`{観点}` の具体化**: 起動するパックの観点リスト（[観点パック定義](#観点パック定義)）をそのまま貼る。
- **ツール権限**: ファイル読み取り・非破壊検証（lint・静的検証・dry-run等）は自由。**実起動も許可**する。ただし破壊的副作用（ファイル変更・外部送信・不可逆操作）を伴う実行が必要な場合は、実行せずユーザーに確認する。

### Step 3: 結果を集約して出力する

- 各レビュアーの出力を、パックごとに[出力フォーマット](#出力フォーマット)で提示する。
- メインAgentは判定区分を改変しない。レビュアーが「該当なし」とした区分はそのまま「該当なし」と示す。
- **報告のみ。** ここで修正・タスク化はしない（必要なら呼び出し側が別途行う）。

## 観点パック定義

2パックは「外部基準（実機・指定仕様）を要するか否か」で分ける。内的整合は常に適用でき、外的妥当は基準・実機がある範囲で適用する。

### 内的整合パック

外部基準を要さず、対象だけで自己完結に判定できる観点。

- **構造**: 構成・粒度・段階的開示が妥当か。肥大・断片化していないか。
- **適用条件の明確さ**: 起動条件・スコープ・前提が明示され、曖昧・過剰・不足でないか。
- **自己矛盾の不在**: 記述間・節間・例と本文の間で矛盾していないか。
- **記述の一貫性**: 用語・形式・語調・命名が揃っているか。
- **過不足**: 欠けている必須要素、冗長・重複している要素。

### 外的妥当パック

実機・指定基準があって初めて判定できる観点。

- **動的検証**: 実際に動かし、宣言どおり動くか（非破壊を優先。破壊的副作用を伴うものは実行せず、その旨を報告）。
- **事実・計算の正しさ**: 記載の事実・数値・コマンド・参照が正しいか。
- **指定基準への準拠**: ユーザー指定の検証基準（仕様）に適合するか。基準が無ければ言語/フレームワーク標準のみに照らす（ドメイン固有仕様は推測しない）。

## レビュアープロンプトテンプレート

サブAgentへ渡すプロンプトは以下で固定する。可変部は `{対象}`・`{検証基準}`・`{観点}` の3つのみで、それ以外を改変しない。`{観点}` には起動するパックの観点リスト（[観点パック定義](#観点パック定義)）を貼る。検証基準が無い場合は `{検証基準}` を「指定なし。対象から自明な言語/フレームワーク標準のみを用いる」に置き換える。

対象: {対象}

# 禁則
- 忖度しない。問題が無ければ各区分で「該当なし」と明言する。問題をでっち上げない。
- 与えられていない設計意図を前提にしない。意図の確認が必要になったら ask 区分で問う。
- 実機確認できるものは確認する。非破壊の手段を優先し、破壊的副作用（ファイル変更・外部送信・不可逆操作）を伴う実行はせず、その旨を報告する。

# 観点
{観点}

# 判定区分（各findingに必ず1つだけ付ける）
- must: 直さないと承認できない。正しさ・安全・仕様/原則違反は、ブロックの確信が持てなくても must に上げる（迷えば must）。
- ask: 回答が要る純粋な疑問。仕様・意図の確認が必要なものはここ。
- imo: 軽微な見解・提案。修正かタスク化の検討対象。
- fyi: 情報共有。アクション不要。
- nits: 重箱の隅。無視可。
- nr: 急がない将来課題。タスク化の検討対象。

各findingは「[対象箇所] 指摘内容（実機確認した場合は結果も）」の形で記す。

## 使用例

**スキルを公開前にフラットレビューする（両パック）:**

```
対象: .apm/skills/foo/SKILL.md（実行可能なスクリプト同梱）
→ 観点パック未指定・実行可能 → 内的整合・外的妥当の2体を並列起動
→ 検証基準: ユーザーがskill-creator仕様を明示 → 外的妥当パックに渡す
```

**設計ドキュメントをレビューする（内的整合のみ）:**

```
対象: docs/design.md
→ 非実行物 → 内的整合パック1体のみ
→ 検証基準なし
```

**PR差分をレビューする:**

```
対象: git diff main...HEAD
→ 実行可否に応じてパックを選ぶ
```
